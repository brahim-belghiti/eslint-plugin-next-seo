## What V0.4 adds (5 rules)

All five are metadata-shape correctness rules. They extend the existing pattern; **no new utilities are required**.

| # | Rule | Severity in recommended | What it catches |
|---|------|-------------------------|-----------------|
| 1 | `metadata-title-length` | `warn` | Static `title` over 60 chars (truncated in SERPs) |
| 2 | `metadata-description-length` | `warn` | Static `description` over 160 chars (truncated in SERPs) |
| 3 | `valid-twitter-card` | `error` | `twitter.card` not in `{summary, summary_large_image, app, player}` |
| 4 | `valid-openGraph-type` | `error` | `openGraph.type` not in the OG type whitelist |
| 5 | `metadata-keywords-shape` | `error` | `keywords: "a, b, c"` (string) instead of `["a", "b", "c"]` (array) |

## Conventions for new rules

Mirror the shape of `src/rules/og-image-in-metadata.ts` and its test file `tests/rules/og-image-in-metadata.test.ts`. They are the cleanest model for a new direct-style rule. Specifically:

1. **Visitor**: register `Program(program)`, call `findMetadataExport(program)`, bail on `!result || !result.object`.
2. **No false positives**: if the value being checked is anything other than a string literal (or array literal where applicable), the rule MUST stay silent. Variable references, function calls, computed values — all silent.
3. **Reuse `findProperty(object, name)` from `src/utils/metadata.ts`** to read `title`, `description`, `twitter`, `openGraph`, `keywords`. It already handles both Identifier keys and string-literal keys.
4. **Message IDs**: one per failure mode. Use `data: { ... }` for placeholders — read `src/rules/no-empty-metadata-fields.ts` and `src/rules/valid-jsonld-type.ts` for examples.
5. **Tests**: every rule needs ~6 valid + 4 invalid cases. Cover both `metadata` object form and `generateMetadata` function form (sync function declaration + async function declaration + arrow). Also cover the silence cases (variable reference, non-string-literal).
6. **TypeScript strict**: assume `noUncheckedIndexedAccess`. Use `!` only when you've narrowed via `if`.
7. **Always run `./node_modules/.bin/vitest run tests/rules/<rule>.test.ts` after writing each rule.** All tests must pass before proposing the commit.
8. **No emojis. No comments unless the why is non-obvious.**
9. **Use Write for new files, Edit for existing.** Don't write a new file when an Edit will do.

## Rule specifications

### Rule 1 — `metadata-title-length`

- **Severity**: `warn` in recommended config (do NOT use `error` for length-based rules — they're aspirational, not correctness).
- **Type**: `"suggestion"` in `meta.type`.
- **What it checks**: `metadata.title` value when statically resolvable to a string.
  - String literal: `title: "About"` → length is `4`.
  - Object form `title: { default: "..." }`: check `default`'s length. (`absolute: "..."` is also valid Next.js — check that too.)
  - `title: { template: "..." }`: ignore the template (the substituted value, not the template, is what renders).
  - Anything else (variable, expression, function): silent.
- **Threshold**: 60 characters. Define as a top-level `const TITLE_MAX = 60;`.
- **Message ID**: `titleTooLong`
- **Message text**: `` "Metadata `title` is {{length}} characters; titles over 60 typically get truncated in search results." ``
- **Report node**: the offending value node (the string literal). For `{ default: "..." }` form, report on the `default`'s value node.
- **Tests**:
  - Valid: short title (string), short title via `{ default }`, short title via `{ absolute }`, no title at all (different rule's job), title set to a variable (silent), template-only title (silent).
  - Invalid: 70-char title (string), 70-char title via `{ default }`, 70-char title via `{ absolute }`, generateMetadata returning a long title.

### Rule 2 — `metadata-description-length`

- **Severity**: `warn`.
- **Type**: `"suggestion"`.
- **What it checks**: `metadata.description` value when a string literal.
- **Threshold**: 160 characters. `const DESCRIPTION_MAX = 160;`
- **Message ID**: `descriptionTooLong`
- **Message text**: `` "Metadata `description` is {{length}} characters; descriptions over 160 typically get truncated in search results." ``
- **Report node**: the description's value node.
- **Tests**: same shape as title-length tests, simpler (description doesn't have `default`/`absolute`/`template` variants — it's just a string).

### Rule 3 — `valid-twitter-card`

- **Severity**: `error`.
- **Type**: `"problem"`.
- **What it checks**: `metadata.twitter.card` when a string literal.
  - Whitelist: `["summary", "summary_large_image", "app", "player"]`.
  - If value is a string literal not in the whitelist → report.
  - If `twitter` is missing entirely → silent.
  - If `twitter.card` is missing → silent. (The Twitter spec defaults to `summary`.)
  - If `twitter` is anything other than an `ObjectExpression` (variable reference, etc.) → silent.
  - If `card` is non-string-literal → silent.
- **Message ID**: `unknownCard`
- **Message text**: `` "Twitter `card` `{{value}}` is not valid. Use `summary`, `summary_large_image`, `app`, or `player`." ``
- **Data**: `{ value }`.
- **Report node**: the card's value node.
- **Tests**: 6+ valid (no twitter, twitter without card, valid card values × 4, twitter as variable), 3+ invalid (bad string value, generateMetadata variant).

### Rule 4 — `valid-openGraph-type`

- **Severity**: `error`.
- **Type**: `"problem"`.
- **What it checks**: `metadata.openGraph.type` when a string literal.
  - Whitelist (define as a Set in `src/utils/og-types.ts`):
    ```
    website, article, book, profile,
    music.song, music.album, music.playlist, music.radio_station,
    video.movie, video.episode, video.tv_show, video.other
    ```
  - Same silence rules as Rule 3 (silent on missing `openGraph`, missing `type`, non-literal value, non-object `openGraph`).
- **Message ID**: `unknownOgType`
- **Message text**: `` "Open Graph `type` `{{value}}` is not a valid Open Graph type." ``
- **Data**: `{ value }`.
- **Report node**: the type's value node.
- **Implementation note**: Put the whitelist in a separate `src/utils/og-types.ts` exporting `OG_TYPES: ReadonlySet<string>`. Tests for the utility are nice-to-have (3-4 cases) but not required.
- **Tests**: 6+ valid, 3+ invalid.

### Rule 5 — `metadata-keywords-shape`

- **Severity**: `error`.
- **Type**: `"problem"`.
- **What it checks**: `metadata.keywords`.
  - If value is a string literal → report (`keywords: "seo, nextjs"` is the bug).
  - If value is an `ArrayExpression` → silent.
  - If anything else → silent.
- **Message ID**: `stringKeywords`
- **Message text**: `` "Metadata `keywords` should be an array of strings, not a single string. Comma-separated keyword strings are deprecated SEO advice." ``
- **Report node**: the keywords value node.
- **Tests**: valid (no keywords, array form, variable reference), invalid (string form, generateMetadata variant returning string).
