## What V0.7 adds — the `init` subcommand

V0.7 closes a UX gap: `require-sitemap` and `require-robots` tell the user "this is missing." V0.7 lets them run a follow-up command to **scaffold** those files based on their actual app/ routes:

```sh
npx next-seo init                        # scaffolds both app/sitemap.ts and app/robots.ts
npx next-seo init sitemap                # sitemap only
npx next-seo init robots                 # robots only
npx next-seo init --base-url https://example.com    # set the production URL
npx next-seo init --dry-run              # print what would be written, don't touch the filesystem
npx next-seo init --force                # overwrite existing files
```

The interesting bit: `init sitemap` walks `app/`, separates **static routes** (no `[param]` segments) from **dynamic routes** (with `[param]`), and pre-populates the sitemap with the static set. Dynamic routes are emitted as commented-out TODO blocks showing where the user would wire in their data fetching.

### Scope of V0.7

In:
- `init` subcommand with sub-targets (`sitemap`, `robots`, both)
- Flags: `--dir`, `--base-url`, `--dry-run`, `--force`
- Route-path conversion utility (filesystem path → URL path)
- Strip route groups (`(name)`) and parallel route slots (`@slot`) when computing URL paths
- Template generation for sitemap.ts (with static/dynamic split) and robots.ts
- Refusal-by-default if target file already exists; opt-in overwrite via `--force`

Out (deferred to later versions):
- Interactive prompts (no stdin reads — keep it scriptable/CI-friendly)
- Custom templates / template-directory option
- `init` for `manifest.ts`, `opengraph-image.tsx`, etc.
- Inferring the `BASE_URL` from `package.json`/`vercel.json`/etc.
- The cross-file check `sitemap-route-mismatch` — separate plan, V0.8.

### Type changes

```ts
// types.ts — extend RouteInfo
export type RouteInfo = {
  filePath: string;
  hasMetadata: boolean;
  urlPath: string;      // "/", "/about", "/blog/[slug]"
  isDynamic: boolean;   // true if urlPath contains "["
};
```

`metadata-coverage` already uses `routes` from the accumulator and only reads `hasMetadata`, so no check rewrites needed. Just extending the shape.

### Route-path conversion rules (`route-paths.ts`)

Given an absolute path to `page.tsx` and the absolute `appDir`:

1. Strip everything up to and including `appDir/`.
2. Strip the trailing `/page.{ts,tsx,js,jsx}`.
3. Split into segments.
4. **Drop** segments matching `^\(.+\)$` (route groups — don't appear in the URL).
5. **Drop** segments starting with `@` (parallel route slots — same).
6. Join the remaining segments with `/`. If empty, return `"/"`. Otherwise return `"/" + joined`.

A path is **dynamic** if any remaining segment matches `^\[.+\]$` (covers `[slug]`, `[...rest]`, `[[...optional]]`).

Examples:
- `app/page.tsx` → `"/"`, static
- `app/about/page.tsx` → `"/about"`, static
- `app/(marketing)/pricing/page.tsx` → `"/pricing"`, static
- `app/@modal/login/page.tsx` → `"/login"`, static
- `app/blog/[slug]/page.tsx` → `"/blog/[slug]"`, dynamic
- `app/blog/[...rest]/page.tsx` → `"/blog/[...rest]"`, dynamic

### Walker change

`scanRoutes` already produces `RouteInfo[]`. Now it should also fill `urlPath` and `isDynamic` using the new utility. No new I/O — purely a derivation step.

### `src/cli/init/sitemap.ts`

```ts
export type SitemapOptions = {
  baseUrl: string;
  staticPaths: string[];   // e.g. ["/", "/about", "/blog"]
  dynamicPaths: string[];  // e.g. ["/blog/[slug]"]
};

export function generateSitemap(opts: SitemapOptions): string;
```

Returns a string — the full content of `sitemap.ts`. Looks like:

```ts
import type { MetadataRoute } from "next";

const BASE_URL = "https://example.com";

const staticRoutes = [
  "/",
  "/about",
  "/blog",
  "/contact",
];

export default async function sitemap(): Promise<MetadataRoute.Sitemap> {
  const staticEntries = staticRoutes.map((route) => ({
    url: `${BASE_URL}${route === "/" ? "" : route}`,
    lastModified: new Date(),
    changeFrequency: "weekly" as const,
    priority: route === "/" ? 1.0 : 0.8,
  }));

  // TODO: dynamic route detected at /blog/[slug]
  // Wire your data source and uncomment:
  // const blogEntries = (await getBlogPosts()).map((p) => ({
  //   url: `${BASE_URL}/blog/${p.slug}`,
  //   lastModified: new Date(p.updatedAt),
  //   changeFrequency: "weekly" as const,
  //   priority: 0.7,
  // }));

  return [
    ...staticEntries,
    // ...blogEntries,
  ];
}
```

Notes:
- The `BASE_URL` constant is at the top so it's easy to find and edit.
- The root route (`/`) emits an empty path-suffix to avoid a trailing slash on the homepage URL.
- For each dynamic path, emit a TODO block. Pick a sensible variable name from the first dynamic segment (e.g. `/blog/[slug]` → `blogEntries`).
- If no dynamic paths exist, omit the TODO sections entirely.

### `src/cli/init/index.ts`

```ts
export type InitTarget = "sitemap" | "robots" | "both";

export type InitOptions = {
  appDir: string;
  baseUrl: string;
  target: InitTarget;
  dryRun: boolean;
  force: boolean;
};

export async function runInit(opts: InitOptions, routes: RouteInfo[]): Promise<number>;
```

Returns an exit code. Behavior:

1. For each enabled target (sitemap and/or robots):
   - Compute the destination path (`<appDir>/sitemap.ts` or `<appDir>/robots.ts`).
   - If destination exists and `!force` and `!dryRun` → print error, set exit code 1, skip.
   - Generate content via the corresponding template generator.
   - If `dryRun` → print `--- destPath ---` followed by content.
   - Else → write to disk and print `✓ Wrote destPath (N entries)` for sitemap, `✓ Wrote destPath` for robots.
2. Print a summary line.
3. Return 0 if all writes succeeded (or were dry-run), 1 if any was skipped due to existing-and-no-force.

### `src/cli/index.ts` — argv changes

Add a new positional command. Pseudocode:

```ts
const { values, positionals } = parseArgs({
  options: {
    dir: { type: "string" },
    "base-url": { type: "string" },
    "dry-run": { type: "boolean", default: false },
    force: { type: "boolean", default: false },
  },
  allowPositionals: true,
});

const command = positionals[0];

if (command === "check") {
  // existing
} else if (command === "init") {
  const subTarget = positionals[1];
  const target: InitTarget =
    subTarget === "sitemap" ? "sitemap" :
    subTarget === "robots"  ? "robots"  :
    subTarget === undefined ? "both"    :
    (() => { throw new Error(`Unknown init target: ${subTarget}`); })();

  const appDir = await findAppDir(process.cwd(), values.dir);
  const routes = await scanRoutes(appDir);
  const baseUrl = values["base-url"] ?? "https://example.com";

  process.exit(await runInit(
    { appDir, baseUrl, target, dryRun: values["dry-run"]!, force: values.force! },
    routes,
  ));
} else {
  console.error("Usage: next-seo check|init [sitemap|robots] [--dir <path>] [--base-url <url>] [--dry-run] [--force]");
  process.exit(2);
}
```

## Conventions for V0.7

1. **Pure functions for templates.** `generateSitemap` and `generateRobots` take options, return strings. No I/O, no fs access. Testable with snapshot-style assertions.
2. **All filesystem work in `runInit`.** Read, write, exists checks all happen there.
3. **No interactive prompts.** Everything via flags. CI-friendly.
4. **Tests in `tests/cli/init/` and `tests/cli/route-paths.test.ts`** mirroring source structure.
5. **No new runtime deps.** Use `node:fs/promises` and `node:path` only.
6. **No emojis. No comments unless the why is non-obvious.**
7. **Use Write for new files, Edit for existing.** Don't write a new file when an Edit will do.
