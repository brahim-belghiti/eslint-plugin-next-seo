## Where the project is (after V0.7)

- Currently on `0.7.0`. Published to npm.
- 17 ESLint rules + companion CLI with 3 checks (`require-sitemap`, `require-robots`, `metadata-coverage`) + `init` subcommand.
- `RouteInfo` carries `{ filePath, hasMetadata, urlPath, isDynamic }`.
- `Accumulator` carries `{ appDir, routes, sitemapFile, robotsFile }`.
- The walker reads the sitemap file's path but not its content.
- The init template uses `staticRoutes.map(...)` — works at runtime but defeats static analysis (a `.map()` is not statically enumerable without symbolic execution).

## What V0.8 adds — `sitemap-route-mismatch`

A fourth CLI check that compares routes scanned from `app/` against URL paths extracted from `app/sitemap.ts`. It surfaces:

- **Missing entries** — a static `app/` route exists but the sitemap doesn't list it (search engines won't discover it).
- **Stale entries** — the sitemap lists a path with no corresponding `page.tsx` (dead link).

Plus an info-level note when the sitemap contains dynamic generation that the parser couldn't enumerate, so the user understands what was and wasn't verified.

### Scope of V0.8

In:
- New pure parser: `parseSitemap(source: string)` → `{ staticPaths, hasDynamicGeneration }`
- Walker reads sitemap content and stores the parse result in the accumulator
- New check: `sitemap-route-mismatch` consuming the parse result + routes
- Update the V0.7 init sitemap template to emit explicit object literals (so its output round-trips through the new check cleanly)
- Wire the check into the CLI dispatch and the reporter
- README documents the new check and the parser's known limits

Out (deferred):
- Symbolic execution of `Array.prototype.map` calls over local const arrays
- Resolving identifiers across imports
- Reading `sitemap.xml` files (only `sitemap.ts/tsx/js/jsx` for now)
- A `--json` output flag (separate, low-effort improvement; can land alongside or after)
- Recommending a route be added to the sitemap (autofix) — not in scope; this is a check, not a fixer

## Sitemap parser specification

### Public surface

```ts
// src/cli/sitemap-parse.ts
export type SitemapAnalysis = {
  staticPaths: string[];
  hasDynamicGeneration: boolean;
};

export function parseSitemap(source: string): SitemapAnalysis;
```

Pure. No I/O. Returns `{ staticPaths: [], hasDynamicGeneration: false }` on parse error rather than throwing — failing closed beats crashing the CLI.

### Walking strategy

1. Parse `source` with `@typescript-eslint/parser` (same options as `walker.ts` uses for `page.tsx`). On error, return the empty result.
2. Find the `ExportDefaultDeclaration`. If absent, return empty.
3. From its declaration, get the function body:
   - `FunctionDeclaration` → `node.body`
   - `ArrowFunctionExpression` with block body → `node.body`
   - `ArrowFunctionExpression` with expression body → wrap as a single `ReturnStatement` for traversal
4. Pre-pass: collect top-level constants (file scope). `const NAME = "literal"` only — no objects, no expressions. Used for resolving template literal interpolations.
5. Traverse the function body. For each node:
   - `Property` with key identifier `url` (or `Literal "url"`) AND parent is an `ObjectExpression` → handle the value:
     - `Literal` with string → push extracted path
     - `TemplateLiteral` → try to resolve all interpolations using the constants map; if every one resolves, push extracted path; if any can't be resolved, set `hasDynamicGeneration = true`
     - Anything else → `hasDynamicGeneration = true`
   - `CallExpression` whose callee is a `MemberExpression` with `property.name === "map"` → `hasDynamicGeneration = true`
   - `SpreadElement` whose `argument.type === "CallExpression"` → `hasDynamicGeneration = true`
6. Return `{ staticPaths, hasDynamicGeneration }`.

### URL → path normalization

```ts
function urlToPath(url: string): string {
  const noHost = url.replace(/^https?:\/\/[^/]+/, "");
  const noQuery = noHost.split(/[?#]/)[0] ?? "";
  if (!noQuery) return "/";
  if (noQuery.length > 1 && noQuery.endsWith("/")) return noQuery.slice(0, -1);
  return noQuery;
}
```

Examples:
- `"https://example.com/about"` → `/about`
- `"https://example.com/"` → `/`
- `"https://example.com/blog/post-1?utm=x#top"` → `/blog/post-1`
- `"/about"` → `/about` (already a path)
- `""` → `/`

### Template literal resolution

Walk the `TemplateLiteral`. For each `quasi` push its `cooked` value. Between quasis, look at the corresponding `expression`:

- `Identifier` whose name is in the constants map → push the resolved value
- Anything else → return `null`, signaling "unresolvable"

Concatenate the parts. If `null` came back, mark `hasDynamicGeneration` and skip this `url`.

### Why scope to the default export's body?

`url` is a common property name. Random object literals elsewhere in the file (config objects, JSDoc-like tags, error payloads) shouldn't pollute the result. Scoping to the function the user actually exports as the sitemap is much closer to "what Next.js will execute."

## `sitemap-route-mismatch` check specification

### Public surface

```ts
// src/cli/checks/sitemap-route-mismatch.ts
import type { Accumulator, Finding } from "../types";
export function run(acc: Accumulator): Finding[];
```

### Logic

Skip entirely when:
- `acc.sitemapFile === null` (no sitemap file — `require-sitemap` reports that separately)
- `acc.sitemapAnalysis === null` (parse failed; don't double-report)

Otherwise, build:
- `sitemapPathSet`: `Set<string>` from `acc.sitemapAnalysis.staticPaths`
- `routeUrlPathSet`: `Set<string>` of all `acc.routes[*].urlPath`
- `dynamicRoutePatterns`: regex array built from `acc.routes` where `isDynamic === true`

Then emit findings:

1. **For each `route` in `acc.routes` with `isDynamic === false`:**
   - If `sitemapPathSet.has(route.urlPath)` → no finding.
   - Else if `acc.sitemapAnalysis.hasDynamicGeneration === true` → no finding (dynamic generation may cover it).
   - Else → emit `error` "Static route `<urlPath>` exists in app/ but is missing from sitemap.ts" with `file: route.filePath`.

2. **For each `path` in `acc.sitemapAnalysis.staticPaths`:**
   - If `routeUrlPathSet.has(path)` → no finding.
   - Else if any `dynamicRoutePatterns[i].test(path)` → no finding.
   - Else → emit `error` "sitemap.ts references `<path>` but no matching page exists in app/" with `file: acc.sitemapFile`.

3. **If `acc.sitemapAnalysis.hasDynamicGeneration === true`:**
   - Emit `info` "sitemap.ts contains dynamic entries; not all paths could be statically verified" with `file: acc.sitemapFile`.

### Dynamic-route regex

```ts
function dynamicPathRegex(urlPath: string): RegExp {
  const pattern = urlPath
    .split("/")
    .map((seg) => {
      if (/^\[\[\.\.\..+\]\]$/.test(seg)) return "(?:.+)?";  // [[...optional]]
      if (/^\[\.\.\..+\]$/.test(seg)) return ".+";            // [...rest]
      if (/^\[.+\]$/.test(seg)) return "[^/]+";               // [slug]
      return seg.replace(/[.*+?^${}()|[\]\\]/g, "\\$&");
    })
    .join("/");
  return new RegExp(`^${pattern}$`);
}
```

Three forms covered: required dynamic (`[slug]`), catch-all (`[...rest]`), optional catch-all (`[[...optional]]`). The optional one is rare but exists in Next.js docs; cheap to support.

## Init-template change (small but important)

Update `src/cli/init/sitemap.ts` so static entries are explicit object literals, not the result of `staticRoutes.map(...)`. Reason: the V0.8 parser can't enumerate `.map()` calls, so the V0.7 template would set `hasDynamicGeneration = true` on its own output and produce only the info note, never useful errors. With explicit literals, every static path the user listed during `init` becomes a verifiable entry.

Old shape (V0.7):
```ts
const staticRoutes = ["/", "/about"];
const staticEntries = staticRoutes.map((route) => ({
  url: `${BASE_URL}${route === "/" ? "" : route}`,
  lastModified: new Date(),
  changeFrequency: "weekly" as const,
  priority: route === "/" ? 1.0 : 0.8,
}));
```

New shape (V0.8):
```ts
const staticEntries = [
  {
    url: `${BASE_URL}`,
    lastModified: new Date(),
    changeFrequency: "weekly" as const,
    priority: 1.0,
  },
  {
    url: `${BASE_URL}/about`,
    lastModified: new Date(),
    changeFrequency: "weekly" as const,
    priority: 0.8,
  },
];
```

A bit longer in output, but every entry is statically extractable AND users can edit per-entry priorities/frequencies without rewriting the array shape. The dynamic TODO blocks are unchanged.

The existing `tests/cli/init/sitemap.test.ts` will fail and need updating — assertions move from `'staticRoutes = ['` to checking individual `{ url: ... }` blocks.

## Conventions for V0.8

1. **Parser is pure.** No I/O. Walker handles file reading.
2. **Failure modes default to silent.** Parse error → empty analysis. Empty analysis → check skipped. Beats crashing or false-positive avalanches.
3. **Tests in `tests/cli/sitemap-parse.test.ts` and `tests/cli/checks/sitemap-route-mismatch.test.ts`** mirroring source structure.
4. **No new runtime deps.** `@typescript-eslint/parser` is already a runtime dep.
5. **No emojis. No comments unless the why is non-obvious.**
6. **Error messages name the path explicitly** — users should be able to grep for it.
